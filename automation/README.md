# Автоматизация обновления отчёта

`dashboard_report.html` изначально собирался вручную: кто-то один раз скормил
всю базу проб в чат с LLM, попросил посчитать метрики и сверстать HTML, и
результат целиком лежит в `для_другого_ПК.zip`. Сам отчёт уже умеет
обновляться *в браузере* без Python (кнопки «📂 Обновить» и «🔧 Отказы» —
человек открывает Excel, сохраняет CSV, перетаскивает файл) — но это всё ещё
ручной процесс. Этот каталог делает то же самое обновление одной командой,
без браузера и без участия человека.

## Как это работает

Вся логика пересчёта (парсинг CSV, метрики WearIdx/ContamIdx/DegradIdx/FP,
z-оценки, скользящие средние Fr3/Fr5m, сборка структуры отказов) уже написана
и оттестирована — она живёт прямо внутри `dashboard_report.html` как чистые
JS-функции без обращений к DOM (`_parseCSV`, `_buildDB`, `_buildPredData`,
`_parseFailCSV`, `_buildFailData`, `_postProcessDB` и т.д.). Вместо того чтобы
переписывать эту логику на Python и рисковать разойтись с оригиналом,
`report_engine.js` — **дословная выдержка** этих функций, запускаемая в Node
вместо браузера. Это гарантирует, что автоматическое обновление даёт
побайтово ту же логику, что и ручное перетаскивание CSV.

Конвейер:

```
Свод_анализов_масла*.xlsm  ──┐
                              ├─▶ export_oil_data.py ──▶ oil_samples.csv, norms.csv, fleet_pct.json
Сводный анализ ПК.xlsx     ──┘                                    │
                                                                    ▼
Сводный анализ ПК/NTE.xlsx ──▶ export_fail_data.py ──▶ oil_failures.csv
                                                                    │
                                                                    ▼
dashboard_report.html (шаблон) ──▶ build_dashboard.js (report_engine.js) ──▶ dashboard_report.html (обновлён)
```

`refresh_report.py` — обёртка над всеми тремя шагами, с автообнаружением
входных файлов (в т.ч. учитывает, что имена с кириллицей в этом репозитории
хранятся то в NFC, то в NFD Unicode-нормализации — `os.listdir` + сравнение
после `unicodedata.normalize` вместо `glob`).

## Файлы

- `report_engine.js` — дословно скопированные чистые функции трансформации
  данных из `dashboard_report.html`. Не редактировать вручную независимо от
  шаблона — если в шаблоне меняется логика `_buildDB`/`_buildFailData`/итд,
  копию здесь нужно обновить тем же куском кода.
- `export_oil_data.py` — сливает один или несколько `Свод_анализов_масла*.xlsm`
  (автоопределяет, в какой строке реальный заголовок — инкрементные файлы
  иногда без строки-группировки), нормализует переносы строк в заголовках
  и в текстовых ячейках (иначе наивный построчный CSV-парсер в
  `report_engine.js` рвёт строки не в тех местах), считает свежую таблицу
  норм (`norms.csv`, аналог `NORMS_CSV_EMBEDDED`) и парковые перцентили P90/P95
  (`fleet_pct.json`, аналог `FLEET_P90`/`FLEET_P95`) — **вместо** того чтобы
  полагаться на статичные константы, один раз посчитанные при ручной сборке
  и с тех пор не обновлявшиеся.
- `export_fail_data.py` — конвертирует лист «Сводные данные» из
  `Сводный анализ ПК/NTE.xlsx` в CSV с колонками, которые ожидает
  `_parseFailCSV`/`_buildFailData` (Дата отказа, Гар#, Узел, Система,
  Воздействие, Описание неисправности).
- `build_dashboard.js` — грузит `report_engine.js`, гоняет через него CSV,
  gzip+base64 кодирует получившуюся базу (`zlib.gzipSync`, формат совместим
  с `DecompressionStream('gzip')` в браузере) и точечно заменяет константы
  `DATA_B64`, `PRED_DATA`, `FAIL_DATA`, `NORMS_CSV_EMBEDDED`, `FLEET_P90`,
  `FLEET_P95` внутри HTML-шаблона, не трогая остальную вёрстку/JS.
- `refresh_report.py` — единая команда для агента/CI: находит входные файлы,
  при необходимости достаёт HTML-шаблон из `для_другого_ПК.zip`, прогоняет
  весь конвейер, пишет `dashboard_report.html`.

## Проверено

`refresh_report.py` прогнан на реальных файлах репозитория: 46 936 проб масла
(слиты `Свод_анализов_масла.xlsm` + `...2026.xlsm`), 369 марок техники,
1 344 единицы, наложены 283 391 событие отказов на 4 313 пар машина|узел.
Результат открыт в headless Chromium (Playwright) — загружается без единой
JS-ошибки, KPI в шапке совпадают с логами сборки, переключение вкладок
работает.

## Известные ограничения / куда копать дальше

- `norms.csv` считается перцентилем по всей истории каждой пары
  (Модель узла|Узел) без выделения выбросов — при малом числе проб (n=1-2)
  p90 вырождается в само значение. В оригинальной статичной таблице то же
  ограничение уже было.
- `Сводный анализ ПК.xlsx` (291 092 записи) содержит ~21% строк без
  классификации Узел/Система/Воздействие («Методика и легенда» в NTE-файле
  прямым текстом называет её «черновой»). `export_fail_data.py` просто
  прокидывает то, что есть — это отдельная задача (NLP-классификация текста
  «Описание неисправности»), не решается пересборкой отчёта.
- Скрипт не выбирает автоматически между `Сводный анализ ПК.xlsx` (весь парк)
  и `NTE.xlsx` (подмножество NTE240/TR100) — по умолчанию берёт ПК; передайте
  `--failures` явно, если нужно другое.
